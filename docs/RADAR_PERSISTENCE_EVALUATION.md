# Bot1 异动雷达持久化方案评估

## 当前结论

异动雷达已完成本地 dry-run 和进程内状态版本，但当前明确禁止生产启用。

项目现有 SQLite 能力主要保存用户、命令日志、币种查询统计和价格快照。`price_snapshots` 是行情事实记录，不包含告警方向、发送状态、冷却截止时间或频道小时配额，也不能在多个触发进程之间原子抢占发送资格。直接把价格快照当作告警幂等状态会造成重复发送或错误限流，因此本次不复用它写入雷达状态。

本次没有新增数据库表、外部服务、生产环境变量，也没有修改 `.env` 或部署配置。

## 本地状态模型

- 冷却键：`coin_id + direction`。
- 冷却值：最近一次明确成功或发送结果 uncertain 的北京时间。
- 小时配额键：北京时间自然小时，例如 `2026-06-22:10`。
- 小时配额值：该小时已明确成功或 uncertain 的条数。
- 明确失败不占冷却和小时配额；dry-run 完全不写状态。

这些状态只存在于单个 Python 进程内。进程重启、休眠恢复、重新部署或另一个实例并发运行时，状态不会共享。

## 生产风险

1. 重启后冷却丢失，同币同方向可能在 2 小时内重复发送。
2. 多实例分别计数，合计可能超过每小时 3 条。
3. 查询、判断和状态写入不是跨进程原子操作，并发触发可能同时通过检查。
4. Telegram 超时只能在当前进程标记 uncertain；重启后无法继续阻止重复。
5. GitHub Actions 临时文件和免费部署实例的本地 SQLite 都不能视为可靠的长期幂等存储。

## 候选持久化方案

### 方案 A：复用现有 SQLite 文件并新增专用表

优点是改动和成本最低，可用事务保存告警记录及小时配额。缺点是当前 Bot1 运行位置的 SQLite 文件可能是临时产物，多实例也需要共享同一文件和可靠文件锁，因此只适合单机、持久磁盘明确可用的部署，不足以直接满足当前生产条件。

建议的专用数据应至少包含：币种、方向、数据时间、触发时间、发送状态、发送时间、小时槽和稳定的幂等键。该结构仅为评估，本次未创建。

### 方案 B：共享 PostgreSQL

可用唯一约束和事务原子处理冷却、小时配额及多实例抢占，最适合生产一致性。但它属于新增外部持久化和环境配置，超出本次授权范围。

### 方案 C：共享 Redis

TTL 和原子计数适合冷却与小时限流，但发送审计和 uncertain 恢复仍需额外设计，也属于新增外部服务，超出本次授权范围。

## 建议与生产准入条件

优先评估方案 B；如果确认 Bot1 始终单实例且具备可靠持久磁盘，可重新评估方案 A。生产启用前至少完成：

1. 明确获批的持久化介质和数据结构。
2. 用事务或等价原子操作同时执行幂等抢占、方向冷却和小时配额检查。
3. 覆盖进程重启、多实例并发、Telegram uncertain 和小时切换测试。
4. 增加可审计日志和人工停用开关。
5. 经 dry-run 观察后，由人工显式解除生产发送硬锁并接入独立调度。

在以上条件完成前，不得把 `RADAR_PRODUCTION_ENABLED` 改为 `true`，不得为雷达增加生产 cron 或内部触发入口。
